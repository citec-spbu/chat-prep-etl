# Git Workflow Standard

Регламент работы с репозиторием **chat-prep-etl**. Обязателен для сохранения чистоты истории.

---

## 1. Правила веток (Branching)

* **Запрет:** Прямые пуши в `main` запрещены.
* **Формат имени:** `{тип}/{описание}` (только строчные буквы, цифры и дефис).
  * `feat/` — новые фичи.
  * `fix/` — исправление багов.
  * `docs/` — документация.
  * `research/` — аналитика и тесты.

*Пример: `feat/telegram-parser`*

---

## 2. Жизненный цикл задачи (Алгоритм)

1.  **Синхронизация:** `git checkout main` && `git pull origin main`.
2.  **Создание ветки:** `git checkout -b type/description`.
3.  **Разработка:** Код → `git add .` → `git commit -m "описание"`.
4.  **Актуализация:** Перед пушем вмерджите свежий `main` в свою ветку: 
    `git merge main`.
5.  **Пуш:** `git push origin {branch-name}`.

---


## 3. Актуализация ветки(перезапись ветки на начало актуального main)
1.  **Синхронизация:** `git checkout main` && `git pull origin main`.
2.  **Переход на ветку:** `git checkout feat/script`
3.  **Отклеиваем комиты от старого коммита и клеим поверх актуального:**`git rebase main`
 Если возникают конфликты: 
Исправьте файлы: `git add .`  && `git rebase --continue`
!!! НЕ ИСПОЛЬЗУЙТЕ `git commit` во время ребейза. Если что-то пошло не так, возрвращаемся с помощью `git rebase --abort` 
4. **Синхронизация:** `git push --force origin <название_вашей_ветки>`

---

## 4. Инструкция по Pull Request (PR)

На GitHub после пуша:
1.  Нажми **"Compare & pull request"**.
2.  Дождись прохождения автоматических проверок (GitHub Actions).
3.  Нажми **"Enable auto-merge"**.
4.  Выбери метод **"Merge commit"** (обязательно для сохранения «петли» на графе).

---

## 5. Безопасность и Лимиты

**Строго запрещено коммитить:**
* Файлы сессий (`.session`) и ключи (`.env`).
* Любые JSON-файлы с данными (кроме `data/samples/`).
* Папки IDE (`.vscode/`, `.idea/`).

Перед пушем всегда проверяйте статус: `git status`.
