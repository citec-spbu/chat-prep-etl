# Git Workflow

Данный документ устанавливает правила работы с репозиторием для обеспечения чистоты истории и стабильности проекта.

## 1. Основные принципы

* **Main.** Прямые пуши в ветку `main` запрещены.
* **Одна задача — одна ветка.** Любое изменение вносится через Pull Request.
* **Автоматизация.** Код вливается только после прохождения CI-проверок.

## 2. Именование веток (Branch Naming)

Имя ветки проверяется автоматически. Используйте следующий формат: `{тип}/{описание-на-английском}`. Описание только с использование цифр и строчных букв, -.

**Допустимые типы:**

* `feat/` — разработка новой функциональности (например, `feat/telegram-parser`).
* `fix/` — исправление багов (например, `fix/json-decode-error`).
* `research/` — проведение тестов и аналитика (например, `research/prompt-eval-accuracy`).
* `docs/` — работа с документацией.

*Пример: `feat/anonymization-logic`*

## 3. Работа с коммитами

Используем **Merge Commits** для сохранения наглядного графа разработки.

* **Сообщения:** Должны быть осмысленными (минимум 5 символов).
* **Содержание:** Один коммит должен решать одну небольшую подзадачу.
* **Граф:** При слиянии на графе должна быть видна «петля» — точка выхода из `main` и точка возврата.

## 4. Жизненный цикл задачи!

1.  **Обновление:** Перейдите в `main` и подтяните изменения: `git pull origin main`.
2.  **Ветка:** Создайте новую ветку: `git checkout -b feat/my-task-name`.
3.  **Разработка:** Пишите код и делайте коммиты.
4.  **Синхронизация:** Если в `main` за это время ушли правки коллег, вмерджите их в свою ветку: `git merge main`.
5.  **Pull Request:** Запушьте ветку и откройте PR на GitHub.
6.  **CI Checks:** Дождитесь прохождения автоматических тестов (Linter + Branch Name Audit).
7.  **Auto-merge:** Нажмите кнопку **Enable auto-merge**. После прохождения проверок код попадет в `main` автоматически.

## 5. Безопасность данных (Критично!)
Согласно требованиям проекта, **запрещено** фиксировать в Git:
* Любые файлы с расширением `.json` (кроме папки `data/samples/`).
* Персональные данные (ФИО, телефоны) в примерах кода.
* API-ключи и конфигурационные файлы `.env`.

Убедитесь, что ваш `.gitignore` настроен правильно перед первым пушем.
